# 项目简介
HDMI Viewer 是一个基于 Windows 的轻量级视频采集与显示工具，用于将外部电脑（如信创/UOS设备）的 HDMI 输出，通过采集卡实时显示在 Windows 窗口中。

本项目主要用于提升双电脑办公场景下的操作连续性，减少传统 HDMI 切换器带来的黑屏和切换中断问题。

# 背景与目标
在双电脑办公环境中（Windows + 信创/UOS），传统方案通常存在以下问题：

+ 使用 HDMI 切换器会导致黑屏和信号重同步
+ 显示器输入切换过程影响工作连续性
+ 多设备切换体验割裂

本项目目标是提供一种替代方案，使信创电脑可以作为 Windows 上的一个实时窗口进行操作，从而提升工作流连续性。

# 系统架构C
本项目支持两种使用模式：

## 直连模式（零延迟）
```java
信创电脑 ── HDMI ── HDMI切换器 ── 显示器
```

特点：

+ 零延迟  
原生画质  
无采集链路

缺点：

+ 切换时存在黑屏  
无法与 Windows 同屏使用

## Viewer 模式（无黑屏）
```java
信创电脑 ── HDMI ── 采集卡 ── USB ── Windows HDMI Viewer ── 显示器
```

特点：

+ 无需切换显示器输入源
+ 无黑屏
+ 信创画面作为 Windows 窗口显示
+ 支持全屏 / 窗口模式

缺点：

+ 存在一定采集延迟
+ 依赖 Windows 主机运行
+ 额外硬件成本

## 双模式切换设计
系统可通过 HDMI 输出端切换器，在两种模式之间自由切换：

+ 直连模式（性能优先）
+ Viewer 模式（体验优先）功能特性
+ HDMI 采集卡实时画面显示
+ 基于 OpenCV 的视频采集
+ WPF 图形界面
+ 支持窗口/全屏显示
+ 低延迟视频渲染（软件级实现）

+ 需要 Windows 主机持续运行
+ UI 功能较基础

# 设计理念
本项目不是为了替代 OBS 或专业视频采集软件，而是针对双电脑办公场景的轻量化解决方案。

核心思想是：

在“直连性能”和“无黑屏体验”之间提供可切换的双模式工作流

# 关键设计决策说明
## 为什么选择 HDMI 采集方案，而不是远程桌面？
虽然远程方案（如 RDP / VNC / Moonlight）可以实现类似效果，但最终未采用，原因如下：

❌ 不采用远程方案的原因

+ 信创/UOS 环境通常存在较强限制（无法安装远控软件）
+ 网络依赖强，存在延迟波动
+ 安全策略可能禁止远程访问
+ 部分场景无法开启或稳定运行远程服务

✔ HDMI 采集方案优势

+ 完全独立于网络环境
+ 不依赖目标系统权限
+ 稳定性高（硬件级信号）
+ 不受系统策略限制

因此选择 HDMI 采集作为最稳定的跨系统显示方案。

## 为什么不只使用 HDMI 切换器作为唯一方案？
HDMI 切换器虽然简单，但存在明显体验问题：

❌ 切换器的问题

+ 切换时会出现黑屏（信号重同步）
+ 显示器重新握手延迟（0.5–3 秒）
+ 切换状态对旁人可见（工作流被打断）
+ 无法实现窗口级融合显示

✔ Viewer 模式的补充价值

Viewer 模式提供：

+ 无黑屏切换体验
+ 信创系统可作为窗口存在于 Windows 中
+ 可实现全屏 / 分屏 / 画中画能力
+ 提升工作流连续性

因此 HDMI 切换器作为“物理直连方案”，Viewer 作为“软件融合方案”并存。

## 为什么采用“双模式架构”？
本项目核心设计不是单一方案，而是支持两种显示模式：

直连模式（零延迟优先）

Viewer 模式（体验优先）

设计原因如下：

**1. 不同场景需求不同**

+ 长时间专注使用 → 直连模式更合适
+ 高频切换操作 → Viewer 模式更高效

**2. 单一方案无法覆盖所有需求**

+ HDMI 切换器无法提供融合体验
+ Viewer 无法提供零延迟体验

**3. 提高系统灵活性**

通过 HDMI 输出端切换器，可在两种模式之间自由切换：

+ 性能优先
+ 体验优先

# 为什么暂时放弃
1. 直连模式即被采集的信创电脑使用的频率高，使用延迟这么高的采集视频采集方案需要一直忍受延迟，且信创电脑上要用到远程桌面，延迟就更高了，如果延迟高到不可忍受，这个方案也就退化为了HDMI切换器的直连方案，优势全无
2. 需要采集卡、切换器的硬件支持，成本高且设备多显得杂乱

# 放弃后的方案是
想办法解决HDMI切换器的延迟问题，也可能可以满足需求
